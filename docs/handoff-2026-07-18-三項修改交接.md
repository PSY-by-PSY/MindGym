## 1. 更換 App icon（手機桌面上的 App 圖示）

### 問題說明
想把 App 在手機桌面上顯示的圖示換成另外一張照片。

### ⚠️ 交接前要準備的東西
- **新的圖片檔**：請向 Grace 拿新照片原檔。最終需要一張 **1024×1024 像素的正方形 PNG，且不能有透明背景**（App Store 規定）。如果原圖不是正方形，要先決定裁切方式。
- 換 App icon **必須重新打包 App 並上架**（走 Xcode build → TestFlight / App Store 審核），不是改網頁就會生效。請預留上架審核時間。

### 修改方向（給工程師）
- icon 來源檔在 `resources/icon.png`。
- iOS 實際使用的 icon 在 `ios/App/App/Assets.xcassets/AppIcon.appiconset/AppIcon-512@2x.png`（單一 1024px 檔案的現代格式）。
- 步驟：把新圖處理成 1024×1024 無透明 PNG → 同時替換上述兩個檔案（`resources/icon.png` 保持來源檔同步）→ 用 Xcode 重新 build → 發新版本。

## 2. 社群貼文翻譯功能：按了沒反應／顯示「翻譯失敗」

### 問題說明
社群貼文下方的「翻譯粉粿」按鈕（介面語言切到英文或簡體中文時出現），按了會轉圈然後顯示「翻譯失敗，請稍後再試」，或看起來沒反應。

### 已經先查過的現況（很重要，直接給工程師省時間）
- 前端功能是完整的：`src/routes/app.community.tsx`（搜尋「翻譯粉粿」，約 1508–1653 行）＋ `src/lib/translate.ts`。
- 後端是 Supabase Edge Function `translate-post`（`supabase/functions/translate-post/index.ts`），**已確認部署且狀態 ACTIVE**，`ANTHROPIC_API_KEY` 密鑰**也已設定**（2026-07-16 更新過）。
- 所以問題不是「沒部署」或「沒金鑰」，最可能的原因：
  1. **`max_tokens: 1024` 太小**（`translate-post/index.ts` 第 50 行附近）：長貼文（例如工作坊的自我敘事有五個欄位）翻成英文很容易超過 1024 tokens，回應被截斷 → JSON 解析失敗 → 回 500 → 前端顯示「翻譯失敗」。
  2. `meterClaude(...)` 計量函式若失敗會直接把整個請求變成 500（沒有 try/catch）。
  3. 模型偶爾回傳帶 Markdown code fence 的 JSON，目前的解析會失敗。

### 修改方向（給工程師）
1. 先到 Supabase Dashboard → Edge Functions → `translate-post` → Logs，重現一次失敗，看實際錯誤訊息（error 都有 `console.error` 記錄）。
2. 依 log 對症下藥，但無論如何建議一併做這三件事：
   - `max_tokens` 從 1024 提高到 4096。
   - `meterClaude` 包 try/catch，計量失敗不應該讓翻譯整個失敗。
   - 解析前先剝掉可能的 ``` code fence。
3. 改完 `npx supabase functions deploy translate-post` 重新部署。

### 可直接複製的 Prompt
```
MindGym 社群的「翻譯粉粿」貼文翻譯功能會失敗（前端顯示「翻譯失敗，請稍後再試」）。

背景：前端在 src/routes/app.community.tsx（搜尋「翻譯粉粿」）與 src/lib/translate.ts，
後端是 Supabase Edge Function supabase/functions/translate-post/index.ts。
函式已部署、ANTHROPIC_API_KEY 已設定，所以請從函式本身的錯誤查起。

請你：
1. 到 Supabase Dashboard 的 translate-post function logs 重現並確認實際錯誤。
2. 在 translate-post/index.ts 做以下修正：
   - max_tokens 由 1024 提高到 4096（長貼文翻譯目前會被截斷導致 JSON 解析失敗）。
   - meterClaude 呼叫包 try/catch，計量失敗時只記 log、不要讓翻譯請求變 500。
   - JSON.parse 之前先移除模型可能加上的 Markdown code fence。

4. 驗證：介面語言切到 English，對一篇「長貼文」（例如工作坊/找尋真實自我的貼文）
   按「翻譯粉粿」，要能成功顯示英文翻譯；再切回「顯示原文」要正常。
```

### 驗收方式
- 介面切到英文與簡體中文，各對一篇**長貼文**和一篇短貼文按「翻譯粉粿」，都能翻譯成功並可切回原文。

## 3. 捲到頂／底時，導覽列與內容之間出現縫隙（iOS 回彈效果）

### 問題說明
在首頁或其他頁面，捲到最上面再繼續往下拉時，頂部導覽列和內容之間會拉出一條縫隙；捲到最底再往上拉時，底部導覽列和內容底端也會出現空隙。希望關掉這個效果，讓畫面捲到頂／底就直接停住。

### 已經先查過的現況（很重要）
這是 iOS 原生的「橡皮筋回彈（rubber-band）」：頁面被拉出彈性位移，而上下導覽列是固定釘在畫面上的，兩者之間就露出縫隙。重點是：

- **修正程式其實已經寫好了**：`ios/App/App/AppDelegate.swift` 的 `MainViewController` 裡有 `webView?.scrollView.bounces = false`（關閉原生回彈），CSS 端 `src/index.css` 也已加 `overscroll-behavior: none`。這段程式是 **2026-07-14 才加入**的。
- 但這屬於「iOS 原生殼」的修改，**必須重新打包 App 上架才會生效**（這個 App 的內容是連到線上網站、改網頁就即時更新，但原生殼的程式不是）。目前使用者裝的版本應該是 7/14 之前打包的，所以還會看到縫隙。
- **結論：這一項不用寫新程式，只需要重新打包上架。可以和第 1 項（換 App icon）併在同一次版本一起送審。**

### 可直接複製的 Prompt
```
MindGym iOS App 捲到頁面頂/底再繼續拉時，固定的上下導覽列和內容之間會出現縫隙
（iOS 橡皮筋回彈效果）。

已確認 repo 裡的修正已經存在：ios/App/App/AppDelegate.swift 的 MainViewController
有 webView?.scrollView.bounces = false（2026-07-14 加入），但目前上架的 App 版本
應該是在這之前打包的。

請你：
1. 確認目前 TestFlight / App Store 上的版本 build 時間是否早於 2026-07-14。
2. 若是，用 Xcode 以目前的 main 分支重新打包（可與更換 App icon 併同一版），
   送 TestFlight。
3. 在新版 App 驗證：
   - 首頁與其他頁面捲到頂/底再拉，畫面直接停住，導覽列與內容之間不再出現縫隙。
   - 社群頁的「下拉重整」仍要正常（它是自訂觸控事件實作、不依賴回彈，理論上不受
     影響，但務必實測）。
```

### 驗收方式
- 新版 App 中，任何頁面捲到頂／底再拉都不再出現縫隙。
- 社群頁下拉重整功能正常。

### 附註
如果是在**手機瀏覽器（Safari）開網頁版**看到的縫隙，那是瀏覽器本身的回彈行為，上面的修正只對 App 生效；若網頁版也要處理，請另外跟工程師討論（效果不一定能完全關掉）。
